## 拆解方法论(参考艾尔登法环文章结构)

> 方法论参考文章:`参考资料/艾尔登法环战斗系统拆解.md`(机核网 151854)
> 应用原则见下方"Mavis 输出原则"

### 4 步通用拆解思路
1. **玩透 + 框图**
   - 玩透游戏战斗系统,画出战斗循环框图
   - 框图按 3 类要素绘制:**玩家行为 / 战中资源 / 敌人状态**
2. **主被动反应频率**
   - 分析游戏的战斗交互频率强度
   - 想要传达的战斗体验是什么
   - 判断游戏内是否可感知设计预期
3. **单次操作的操作精度**
   - 确定核心操作乐趣所在,判断是否合理
   - 子维度:**指令复杂度 / 空间精度 / 时间精度**
4. **精通维度 / 精通循环**
   - 判断游戏是否能形成精通循环
   - 带来成长感和成就感
   - 子维度:**策略与判断 / 技术和熟练度**

### 发散讨论维度(按游戏需要选择)
- **难度选择**:是否提供难度?提供后是否破坏整体体验?
- **剧情选择**:主控角色非玩家创造时,重大剧情节点的选择如何处理?
- 其他:可按游戏特性扩展

### multi-lang-email 时区硬编码规则 (2026-06-04)
Type: preference

在 multi-lang-email 技能(MH 运营邮件翻译)中,**时区处理必须按语言硬编码,无视模板里的时区标记**:
- 12 种语言(EN/CN/RU/DE/FR/IT/TH/ES/PT/TR/PL/JP)保持 UTC
- ID 加 7h 标 WIB
- TW 加 8h 标 (UTC+8)
- KR 加 9h 标 (UTC+9)

**即使原模板里 TW/KR 用了 `UTC` 标记(不是 UTC+8/UTC+9),也按 +8h/+9h 强制换算**。
原模板的日期数字格式(YYYY/M/D vs M/D/YYYY vs YYYY年M月D日 等本地化习惯)仍沿用,但**时区标记是硬编码的**。

触发条件:用户说"按本地时区"时,**实际意思是"按区域硬编码",不是"按模板原文"**。
反例:曾把规则写成"看模板里写的时区标记"——被纠正为硬编码。

更广泛的偏好:用户偏好**标准化、可预测的输出**,而不是"灵活适配原文"。在设计规则时先问"硬编码 + 强制"还是"灵活 + 适配",不要默认选后者。
